1. KVCacheManager and Related Classes

KVCacheManager

  • hash_block_size
    Size used for computing block hashes (prefix caching). Different KV cache groups may have larger actual block sizes that are multiples of this.

  • get_computed_blocks(request)
    Call KVCacheCoordinator.find_longest_cache_hit() if prefix caching enabled, else nothing.

  • allocate_slots(request, num_new_tokens, num_new_computed_tokens, ...) called when scheduling waiting or running requests, where

    • computed: blocks allocated before current scheduling
    • new_computed: blocks that hit the prefix cache
    • new: blocks that need to be allocated
    1. Computes the number of new required blocks.
    2. Increments reference counts for cached blocks.
    3. Allocates new blocks from the free queue (or eviction queue) by calling allocate_new_blocks on the coordinator.
    4. Call coordinator’s cache_blocks().
    5. Returns a KVCacheBlocks object.

    Note: Once a request enters the running queue, it is no longer considered for prefix caching (num_new_computed_tokens = 0).

KVCacheCoordinator – base coordinator managing multiple single-type managers.

  • KVCacheCoordinatorNoPrefixCache – variant without prefix caching.

    • get_num_blocks_to_allocate() – routes to each single-type manager.
    • allocate_new_blocks(request_id, num_tokens) – routes to each manager.

  • HybridKVCacheCoordinator – used when enabling prefix caching and there are two KV cache group types (e.g., one for full attention, one for Mamba).

    • cache_blocks(self, num_computed_tokens)

BlockPool (inside the coordinator)

  • blocks: list[KVCacheBlock] – all blocks in the pool.
  • null_block – a special block representing null.
  • hash_block_size – same as manager’s hash block size.
  • FreeKVCacheBlockQueue – a doubly‑linked list of free blocks.
  • cached_block_hash_to_block: BlockHashToBlockMap – a dictionary mapping BlockHashWithGroupId to dict[int, KVCacheBlock] (multiple blocks with same hash due to eviction).
  • get_cached_block(): hash lookup in cached_block_hash_to_block.
  • cache_full_blocks(): Maintain cached_block_hash_to_block, where the hashes have been computed by the Request object.

KVCacheBlock

  • Attributes: block_id, block_hash, ref_cnt, and pointers for linked lists.

KVCacheBlocks

  • blocks: tuple[Sequence[KVCacheBlock]] - A tuple where each element corresponds to a KV cache group, and inside each group there is a sequence of blocks (tokens). blocks[i][j] refers to the j‑th block of the i‑th group.

SingleTypeKVCacheManager

  • req_to_blocks: dict[req_id: str, list[KVCacheBlock]] – maps request IDs to their allocated blocks.
  • num_cached_block: dict[str, int] – number of cached blocks per request.
  • get_num_blocks_to_allocate() – computes required blocks for a request.
    • cdiv(num_tokens, block_size) by default or for FullAttentionManager.
    • MambaManager: num_tokens += block_size * num_speculative_blocks if spec decoding; otherwise default.
    • SlidingWindowManager:
  • allocate_new_blocks(): calculate (again) the number of new blocks needed, calls BlockPool.get_new_blocks(), and stores the allocated blocks in the manager’s req_to_blocks mapping.
  • cache_blocks(self, request, num_tokens): call BlockPool.cache_full_blocks.
  • remove_skipped_blocks(): relies on get_num_skipped_tokens().

2. Initialization of KV Caches

EngineCore._initialize_kv_caches

  1. Retrieve the kv_cache_specs.
  2. Generate the configuration via get_kv_cache_configs.
  3. Call initialize_from_config, which calls GPUModelRunner.initialize_kv_cache.

GPUModelRunner.get_kv_cache_spec

  • Enumerates all attention layers.
  • For each layer, calls Attention.get_kv_cache_spec.
  • Returns a Dict[layer_name: str, AttentionSpec].

get_kv_cache_configs (from kv_cache_utils.py) groups specs and produces a unified configuration.

  1. get_kv_cache_groups → _get_kv_cache_groups_uniform_spec / uniform_page_size
    Returns list[KVCacheGroupSpec(layer_names, KVCacheSpec)].
  2. get_kv_cache_config_from_groups
    Returns a KVCacheConfig containing:
    • num_blocks
    • kv_cache_tensors: list[KVCacheTensor(size, shared_by: list[str])] – length equals max group size.
    • kv_cache_groups: list[KVCacheGroupSpec]

Notes:

  • MambaSpec cannot merge with FullAttentionSpec.
  • unify_kv_cache_spec_page_size adjusts block sizes for some specs; for already‑padded Qwen3‑Next Mamba, no operation is performed.

GPUModelRunner.initialize_kv_cache

  • initialize_attn_backend
  • initialize_metadata_builders
  • initialize_kv_cache_tensors:
    • _allocate_kv_cache_tensors(kv_cache_config) - Returns raw tensors per group: dict[layer_name: str, torch.Tensor].
    • _reshape_kv_cache_tensors(kv_cache_config, kv_cache_raw_tensors, kernel_block_size),
      where kernel_block_size is the block size per group. Returns a sliced dictionary dict[layer_name: str, tuple[torch.Tensor]].
    • bind_kv_cache
      1. Sets self.kv_caches to a list of tuples (sorted by layer index).
      2. Configures the KV caches of forward_context[layer_name] (e.g., for Qwen3NextGatedDeltaNet).

3. GPUModelRunner

execute_model(self, scheduler_output, ...):

  1. _update_states(self, scheduler_output)

    • Calls InputBatch.add_request(self, request: CachedRequestState).
    • self._get_valid_sampled_token_count() synchronization

  2. _prepare_inputs

    • self.input_batch.block_table.compute_slot_mapping(req_indices, positions_np)
    • _calc_spec_decode_metadata() handles the speculative tokens in scheduler_output.scheduled_spec_decode_tokens.

  3. _determine_batch_execution_and_padding

  4. _build_attention_metadata

    • Prepares attention metadata for each KV cache group; layers in the same group share metadata.
    • Block tables and slot mappings are already prepared in input_batch.
    • Builds CommonAttentionMetadata, spec_decode_common_attn_metadata: GDNAttentionMetadata.

  5. Other methods: preprocess, model_forward.

sample_tokens():

  1. _sample()
    • rejection_sampler
    • _update_states_after_model_execute.
  2. _propose_draft_token_ids()
    • drafter.prepare_next_token_ids_padded()
    • drafter.prepare_inputs_padded()
    • drafter.propose() - run the MTP layers
    • _copy_valid_sampled_token_count(): trigger copying

BlockTable

  • block_size, kernel_block_size
  • block_table: (max_num_reqs, max_num_blocks_per_req)
  • slot_mapping: (max_num_batched_tokens, )
  • add_row(block_ids: list[int], row_idx: int)
  • compute_slot_mapping(req_indices, positions)

Example parameters:

  • Full attention group: block_size = 32, blocks_per_kv_block = 17, max_num_blocks_per_req = 17 * 8 = 136
  • Mamba group: group_size = 4096, blocks_per_kv_block = 1, max_num_blocks_per_req = 1

MultiGroupBlockTable

  • block_tables: list[BlockTable] – one per group.
  • add_row(block_ids: tuple[list[int], ...], row_idx) – routes to each group’s BlockTable.add_row.

SchedulerOutput (v1/core/shed/output.py)

  • scheduled_new_reqs: list[NewRequestData]
  • scheduled_cached_reqs: CachedRequestData
  • num_scheduled_tokens: dict[str, int]
