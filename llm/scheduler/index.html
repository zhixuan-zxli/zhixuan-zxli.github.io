Scheduler, and KV transfer in disaggregated prefill
2026-06-06 00:00 label vLLM  

1. Scheduler

Scheduler (vllm/v1/core/sched/scheduler.py) is the central resource allocator for vLLM’s v1 engine. Each iteration it decides which requests run, how many tokens each gets, and which KV cache blocks to (re-)use. It owns two queues — self.waiting and self.running — plus a KVCacheManager for block allocation and an optional KVConnectorBase_V1 for cross-node KV transfer. The scheduler has no notion of a distinct “prefill phase” or “decode phase”; every request is simply tracked by num_computed_tokens and num_tokens_with_spec, and the scheduler’s job is to close the gap between them.

1.1. Terminology — token counts on a Request

The scheduling algorithm is intentionally uniform across prefill, chunked prefill, decode, and speculative decoding. That uniformity depends on a handful of per-request counters. All of them live on Request (vllm/v1/request.py):

Field Meaning
num_prompt_tokens Length of the initial prompt. Immutable after admission.
num_tokens len(prompt) + len(output_token_ids) — everything already “committed” to the request. Property, not a stored field.
num_tokens_with_spec num_tokens + len(spec_token_ids) — the target the scheduler tries to catch up to this step.
num_computed_tokens Tokens whose KV has been (or is being) produced. Advanced eagerly by _update_after_schedule using the scheduled count; corrected downward by update_from_output when spec-decode draft tokens are rejected or a KV load fails.
num_new_tokens Per-step delta the scheduler assigns to this request. Starts as num_tokens_with_spec + num_output_placeholders - num_computed_tokens, then is clipped by token_budget, long_prefill_token_threshold, and max_model_len - 1 - num_computed_tokens.
num_output_placeholders Async-scheduling reservation: tokens that have been scheduled but whose sample has not yet come back. Included optimistically in num_computed_tokens; corrected on rejection.
num_external_computed_tokens Tokens claimed from a remote KV cache via connector (P/D disaggregation). Recorded when get_num_new_matched_tokens returns a non-zero count.
num_cached_tokens Tokens with prefix-cache hits (local + external), stamped once when the request first runs.

Loop invariant: each step tries to shrink num_tokens_with_spec - num_computed_tokens. Ordinary decode has gap = 1 (+ any spec drafts); prompt prefill has gap = many, and chunked prefill slices it across steps.

The split _update_after_schedule ⇄ update_from_output is deliberate: the former eagerly advances counters so the next step can plan against the current forward pass (chunked prefill, async scheduling); the latter corrects any over-estimates — spec-decode rejections and failed KV loads — once the model result is known.

1.3. add_request — request enters the system

A Request is constructed from an EngineCoreRequest, pushed onto self.waiting, and registered in self.requests. Status: WAITING. num_computed_tokens = 0; num_cached_tokens = -1 (sentinel — filled in on first scheduling).

1.4. schedule() — the two-pass loop

The main loop runs in two sequential passes, sharing a single token_budget = max_num_scheduled_tokens:

Pass 1 — running requests
Pass 2 — waiting requests   (skipped entirely if any preemption occurred in Pass 1)

Pass 1: running requests

Iterates self.running in queue order. For each request, the number of new tokens to schedule is:

num_new_tokens = request.num_tokens_with_spec \
               + request.num_output_placeholders \
               - request.num_computed_tokens
num_new_tokens = min(num_new_tokens,
                     long_prefill_token_threshold,
                     token_budget,
                     max_model_len - 1 - request.num_computed_tokens)

num_tokens_with_spec includes speculative draft tokens, so this same formula covers ordinary decoding (gap = 1), chunked-prefill continuation (gap = many), and spec decoding (gap = 1 + drafts).

KV block allocation and preemption. Omitted here — see the posts on KV cache / model runner. If allocate_slots cannot find room, the scheduler preempts the lowest-priority running request and retries.

Pass 2: waiting requests

Skipped entirely if any preemption happened in Pass 1 (to avoid starving running requests further). Otherwise, peeks from self.waiting in order. Several checks can defer a request (re-inserted at the head of the queue via skipped_waiting_requests) rather than schedule it:

  • WAITING_FOR_REMOTE_KVS: KV transfer from a remote prefill node is still in flight. _update_waiting_for_remote_kv is called; the request is either woken (status flipped back to WAITING/PREEMPTED) or re-parked.

For requests that survive the checks, the scheduler resolves prefix-cache hits (only for first-touch requests — resumed async-load ones already have num_computed_tokens > 0):

# Local prefix cache hit
new_computed_blocks, num_local_hit = kv_cache_manager.get_computed_blocks(request)

# Remote prefix cache hit (KV connector, e.g. P/D disaggregation)
ext_tokens, load_kv_async = connector.get_num_new_matched_tokens(request, num_local_hit)

num_computed_tokens = num_local_hit + ext_tokens
num_new_tokens = 0 if load_kv_async else request.num_tokens - num_computed_tokens

If load_kv_async is True (the remote KV isn’t here yet), the scheduler still calls allocate_slots — reserving destination blocks — then registers them with the connector via update_state_after_alloc and parks the request as WAITING_FOR_REMOTE_KVS. No forward-pass work is scheduled for the request this step.

Otherwise, after a successful allocate_slots, the request transitions WAITING → RUNNING (or PREEMPTED → RUNNING for a resumption), and the sync-load path lets update_state_after_alloc note the reserved blocks.

1.5. _update_after_schedule

def _update_after_schedule(self, scheduler_output):
    num_scheduled_tokens = scheduler_output.num_scheduled_tokens
    for req_id, n in num_scheduled_tokens.items():
        request = self.requests[req_id]
        request.num_computed_tokens += n         # eager advance
    self.finished_req_ids = set()                # wiped so it doesn't leak into the next output

Things to notice:

  1. Eager, not correct. The advance uses the scheduled count, not what the model actually accepts. Spec-decode rejection unwinds it in update_from_output. Doing the advance here lets the next schedule() see a chunked-prefill request as “already partially computed” without waiting for the model — this is what enables both chunked prefill and async scheduling to keep the GPU busy.
  2. finished_req_ids reset. The set is populated by _free_request during Phase B of the previous update_from_output, read once by this step’s SchedulerOutput, and then cleared here in preparation for the next step.

1.6. KV connector — scheduler-side hooks

At the five hook points marked ▶ in the call stack, the scheduler talks to KVConnectorBase_V1 (role = SCHEDULER). All are pure planning — no tensors, no GPU touching.

get_num_new_matched_tokens(request, num_local_hit) → (ext_tokens, load_kv_async)

update_state_after_alloc(request, blocks, num_external_tokens)

build_connector_meta(scheduler_output) → KVConnectorMetadata

_update_waiting_for_remote_kv(request) → bool (scheduler-internal, connector-driven)

Runs at the top of the Pass-2 body when a request is WAITING_FOR_REMOTE_KVS. Returns True iff the transfer completed since the last step — the check reads self.finished_recving_kv_req_ids, which was populated by the previous update_from_output. On success:

(block_ids,) = self.kv_cache_manager.get_block_ids(request.request_id)
num_computed = min(len(block_ids) * self.block_size, request.num_tokens)
if num_computed == request.num_tokens:
    num_computed -= 1                                # leave at least one token to sample
self.kv_cache_manager.cache_blocks(request, num_computed)
request.num_computed_tokens = num_computed

The request is then flipped back to WAITING (or PREEMPTED for a resumption) and re-enters normal Pass-2 scheduling in the same step, taking the else branch that skips get_computed_blocks — the blocks are already there.

_update_from_kv_xfer_finished(kv_connector_output) (Phase C of update_from_output)

if self.connector is not None:
    self.connector.update_connector_output(kv_connector_output)
for req_id in kv_connector_output.finished_recving or ():
    self.finished_recving_kv_req_ids.add(req_id)     # picked up next schedule()
for req_id in kv_connector_output.finished_sending or ():
    self._free_blocks(self.requests[req_id])         # producer side: release deferred blocks

Two directions:

  • finished_recving on the consumer (decode) node: unblocks _update_waiting_for_remote_kv next step.
  • finished_sending on the producer (prefill) node: releases blocks that request_finished(...) had deferred (i.e. blocks that were held past request completion until the peer confirmed receipt).

request_finished(request, block_ids) → (delay_free_blocks, kv_transfer_params) (called from _free_request)

The connector’s chance to hold on to KV blocks past request finish. Returning delay_free_blocks=True skips the immediate _free_blocks; the blocks are freed later when finished_sending reports the peer has consumed them. kv_transfer_params (block ids, cluster id, host:port, DP rank, …) is shipped back with the EngineCoreOutput so the decode side can locate the source.

1.7. update_from_output(scheduler_output, model_runner_output)

Three phases: fault handling, per-request reconciliation, post-loop bookkeeping.

Phase B — Per-request reconciliation. For each (req_id, n_sched) in scheduler_output.num_scheduled_tokens:

  1. Spec-decode correction. If draft tokens were scheduled for this request:

    num_draft    = len(scheduled_spec_token_ids)
num_accepted = len(generated_token_ids) - 1        # sampler emits accepted + 1 bonus
num_rejected = num_draft - num_accepted
if request.num_computed_tokens > 0:
    request.num_computed_tokens     -= num_rejected
if request.num_output_placeholders > 0:
    request.num_output_placeholders -= num_rejected

    This undoes the over-advancement that _update_after_schedule did with the (optimistic) scheduled count.

  2. Append & stop-check via _update_request_with_output:

    for tok in new_token_ids:
    request.append_output_token_ids(tok)
    if check_stop(request, self.max_model_len):
        stopped = True
        del new_token_ids[num_new:]                # trim tokens after the stop
        break

    check_stop inspects EOS, max_tokens, stop_token_ids, stop_str, and max_model_len.

  3. On stop, _free_request:

    delay_free_blocks, kv_xfer_params = self._connector_finished(request)
self.encoder_cache_manager.free(request)
self.finished_req_ids.add(request.request_id)     # → next SchedulerOutput.finished_req_ids
if not delay_free_blocks:
    self._free_blocks(request)                    # kv_cache_manager.free + del self.requests[id]

    kv_xfer_params is threaded back into EngineCoreOutput for this request so the frontend can pass it to the decode side.

  4. Emit EngineCoreOutput for the client: new tokens, logprobs, pooler output, kv_transfer_params, finish reason, num_cached_tokens, structured-output events.

Phase C — Post-loop.

  • Remove stopped requests from self.running / self.waiting.
  • _update_from_kv_xfer_finished(kv_connector_output) — see §1.6.
  • Drain KV cache events from both the KV cache manager and the connector; publish a KVEventBatch on self.kv_event_publisher.
  • Aggregate finished-request sets per client, attach SchedulerStats + SpecDecodingStats + KVConnectorStats, and return dict[client_index → EngineCoreOutputs].

1.8. Disaggregation lifecycle (scheduler view)

Prefill node. A waiting request is scheduled with load_kv_async=False and its connector metadata built. When the prefill finishes, request_finished returns delay_free_blocks=True; blocks are kept alive until finished_sending arrives from the peer via _update_from_kv_xfer_finished.

Decode node (three rounds):

  • Round 1 — schedule(): waiting request gets ext_tokens > 0 and load_kv_async=True from get_num_new_matched_tokens. allocate_slots reserves destination blocks; update_state_after_alloc registers them with the connector; status flips to WAITING_FOR_REMOTE_KVS. build_connector_meta packages the pending pull; execute_model kicks off start_load_kv on the worker.
  • Round 2 — update_from_output: _update_from_kv_xfer_finished sees the request id in kv_connector_output.finished_recving and puts it in self.finished_recving_kv_req_ids.
  • Round 3 — schedule(): Pass 2’s _update_waiting_for_remote_kv finds the request in finished_recving_kv_req_ids, calls cache_blocks, sets num_computed_tokens = min(len(blocks) * block_size, num_tokens) - (0 or 1), and flips status back to WAITING. Same step, the else-branch schedules only the residual tokens beyond the received KV prefix.

2. Connector

In standard (aggregated) inference, every GPU does both prefill (processing the full prompt) and decode (generating one token at a time). The problem is that these two phases have very different compute profiles: prefill is compute-bound (long sequences), decode is memory-bandwidth-bound (single tokens). Running them together causes head-of-line blocking and GPU underutilization.

Disaggregated prefill splits these into separate nodes. A prefill node runs the full forward pass over the prompt, then ships its KV cache to a decode node, which starts autoregressive generation from that already-computed state.

2.1. Connector Interface: KVConnectorBase_V1

vllm/distributed/kv_transfer/kv_connector/v1/base.py

The abstract base class. Every connector implements two distinct sides:

Scheduler side (runs in the scheduler process, no tensors, pure planning):

Method Purpose
get_num_new_matched_tokens(request, computed) Tell the scheduler how many tokens can be claimed from a remote prefill
update_state_after_alloc(request, blocks, n) Record which blocks were allocated for incoming KV
build_connector_meta(scheduler_output) Package per-step metadata to ship to workers
request_finished(request, block_ids) Optionally defer block freeing until async send completes

Worker side (runs in the GPU worker process, operates on tensors):

Method Purpose
start_load_kv(forward_context) Begin async KV receives (consumer)
wait_for_layer_load(layer_name) Sync point before using a received layer’s KV
save_kv_layer(layer_name, kv_cache, attn_metadata) Async-send a layer’s KV (producer)
wait_for_save() Sync point after all layers are forwarded
get_finished(...) Report which requests finished sending/receiving

The scheduler-side / worker-side split mirrors vLLM v1’s process boundary: the scheduler manipulates block tables and request queues, while workers touch GPU memory.

2.2. Scheduler Integration

vllm/v1/core/sched/scheduler.py

The scheduler instantiates the connector with KVConnectorRole.SCHEDULER and calls it at three points per scheduling step:

  1. Token counting — get_num_new_matched_tokens() lets a consumer claim up to len(prompt) - 1 tokens as “already computed” (coming from the prefill node), avoiding redundant recomputation.
  2. Metadata building — after scheduling decisions, build_connector_meta() packages which requests need sends/receives this step.
  3. Completion — request_finished() lets the connector optionally defer freeing KV blocks until async sends complete.

The scheduler also processes worker feedback: when finished_recving reports a request is fully loaded, it transitions that request from WAITING_FOR_REMOTE_KVS → ready to schedule for decode.

2.3. Worker Integration

vllm/v1/worker/kv_connector_model_runner_mixin.py vllm/attention/utils/kv_transfer_utils.py

KVConnectorModelRunnerMixin wraps each forward pass:

connector.bind_connector_metadata(metadata)
connector.start_load_kv(forward_context)   # kick off async recvs
try:
    # ... model forward pass ...
finally:
    connector.wait_for_save()              # drain async sends
    output.finished_sending, output.finished_recving = connector.get_finished(...)
    connector.clear_connector_metadata()

Each attention layer is wrapped by maybe_transfer_kv_layer (a decorator in kv_transfer_utils.py):

connector.wait_for_layer_load(layer_name)  # wait for this layer's KV
result = attention_forward(...)            # use received KV or compute fresh
connector.save_kv_layer(layer_name, ...)   # send this layer's KV

This per-layer interleaving is the intended design: KV sending and the model forward pass can overlap in time, hiding transfer latency.

2.4. Async KV Transfer Lifecycle (Decode Node)

The actual transfer does not start when request.status = WAITING_FOR_REMOTE_KVS is set. That only parks the request. The transfer is kicked off on the next execute_model call.

Scheduler step N:
  get_num_new_matched_tokens()        -> (N tokens, async=True)
  kv_cache_manager.allocate_blocks()  # reserve destination blocks
  connector.update_state_after_alloc()# register blocks with connector
  request.status = WAITING_FOR_REMOTE_KVS  # park the request
  build_connector_meta()              # serialize pending ops into metadata

execute_model (step N):
  connector.bind_connector_metadata() # load the metadata
  connector.start_load_kv()           # << transfer actually starts here
  [GPU forward pass for other requests] # overlaps with the transfer
  connector.get_finished()            # poll: which transfers completed?

Scheduler step N+1 (or later):
  _update_waiting_for_remote_kv()     # sees request in finished_recving_kv_req_ids
  kv_cache_manager.cache_blocks()
  request.num_computed_tokens = N     # now > 0
  request.status = WAITING            # re-enters scheduling via the else-branch

The else branch in scheduler.py:548 is only reached on the second scheduling attempt after num_computed_tokens has been written by _update_waiting_for_remote_kv. It skips get_computed_blocks entirely — the blocks are already allocated and cached — and schedules only the residual tokens beyond the received KV prefix.

2.5. LLMDataDist

Class Role
PrefillConnectorScheduler Scheduler side on prefill node
PrefillConnectorWorker Worker side on prefill node
DecodeConnectorScheduler Scheduler side on decode node
DecodeConnectorWorker Worker side on decode node

Bulk transfer, not per-layer. The per-layer interleaving methods — wait_for_layer_load, save_kv_layer, and wait_for_save — are all no-ops in LLMDataDistConnector. Instead, the entire KV cache is transferred as a single bulk operation inside start_load_kv. On the decode side, DecodeConnectorWorker.start_load_kv() submits one ThreadPoolExecutor job per request; each job calls datadist_manager.pull_kv(), which issues an RDMA pull over RoCE (via the llm_datadist C++ library). Because the transfer is not interleaved with the forward pass, the decode GPU idles during the transfer step rather than overlapping work.

Scheduler-side flow (decode node). DecodeConnectorScheduler.get_num_new_matched_tokens() returns (len(prompt) - 1, True) for any new decode request, telling the scheduler to treat the entire prompt as externally computed and request an async load. After kv_cache_manager.allocate_slots(), update_state_after_alloc() records the request in _reqs_need_recv. build_connector_meta() then converts that list into a DatadistConnectorMetadata containing, per request:

  • local_block_ids — where to write the pulled KV on the decode node
  • remote_block_ids / remote_host / remote_cluster_id — where to read from on the prefill node
  • kv_transfer_params — additional routing info (DP rank, PP partition)

Block lifetime on the prefill node. PrefillConnectorScheduler.request_finished() returns delay_free_blocks=True, so the scheduler does not immediately release the prefill’s KV blocks. Instead, PrefillConnectorWorker keeps them alive until the decode node confirms receipt via a ZMQ message. The maximum hold time is BLOCK_RELEASE_DELAY = 600 s.

End-to-end sequence of LLMDataDist

  1. Prefill worker register_kv_caches → native runtime pins memory + advertises BlocksCacheKey(cluster_id, model_id); listener already up from init.
  2. Request finishes prefill → scheduler returns kv_transfer_params (block ids + cluster id + host:port) to decode via the vLLM control plane.
  3. Decode update_state_after_alloc stores it; build_connector_metadata hands it to decode worker.
  4. Decode worker _read_blocks → datadist_manager.pull_kv → (link on demand via register_link if needed) → cache_manager.pull_blocks(...) — RDMA READ happens here, entirely native.
  5. Decode ZMQ-pushes the remote_request_id back to prefill’s get_pulled_kv_req_list (line 495-511).
  6. Prefill’s get_finished matches it against requests_finish_time and reports the send as done, freeing blocks.
